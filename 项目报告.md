# 编译原理课程项目报告

## 1. 项目背景

本项目基于 SysY 语言实现一个面向 Koopa IR 和 RISC-V 的编译器。在完成基础前端、IR 生成和后端代码生成后，项目重点放在两类扩展上：

项目仓库：

- GitLab：https://gitlab.eduxiji.net/pku2400013064/dev-sysy
- GitHub：https://github.com/Cai-bird-one/dev-sysy

1. 语言与 IR 扩展：在 SysY 中加入 `tensor` 基本类型，并设计非注释形式的 Koopa tensor dialect，用来保存 tensor 形状和操作关系。
2. 性能优化：围绕 Koopa IR 中端优化、RISC-V 指令选择、窥孔优化和寄存器分配进行优化，并为显式 tensor 矩阵乘法实现专门 lowering。

Tensor 扩展的目标不是把普通数组循环自动猜成矩阵运算，而是让矩阵语义显式进入 IR。这样可以避免误判普通二维数组程序，也方便在固定形状 tensor 上做更激进的矩阵乘、bias、ReLU、转置视图融合优化。

## 2. 项目设计

编译器命令行接口如下：

```sh
compiler (-koopa|-riscv|-perf) <input-file> -o <output-file>
```

主要输入输出：

- `-koopa input.sy -o output.koopa`：SysY 源码输入，输出 Koopa IR。
- `-riscv input.sy -o output.s`：SysY 源码输入，输出普通 RISC-V 汇编。
- `-perf input.sy -o output.s`：SysY 源码输入，经过优化后输出 RISC-V 汇编。
- `-perf input.koopa -o output.s`：直接输入 Koopa IR，便于测试 tensor dialect 和中端优化。

整体流程：

1. Lexer 将源码切分为 token，其中 `tensor` 被识别为关键字。
2. Parser 按 LL(1) 文法构造语法树，`tensor` 和 `int` 一样作为 `BType`。
3. Koopa 生成阶段把 SysY AST 翻译为 Koopa IR；tensor 声明会生成普通数组存储和 `tensor ...` 形状元信息。
4. `IrOptimizer` 解析 Koopa IR，构造 `IrModule`，其中 `IrModule::tensor` 保存 tensor 信息。
5. `buildPerfPipeline()` 反复运行中端 pass，直到不再变化或达到迭代上限。
6. RISC-V 后端解析优化后的 Koopa，进行寄存器分配、指令发射和窥孔优化。

当前主要 pass 如下：

| Pass | 作用 |
| --- | --- |
| `TensorDialectOptimizationPass` | 规整 tensor 元信息，融合 `matmul + bias + relu`，吸收 `transpose` 视图，删除冗余 tensor op |
| `TensorMatmulLoweringPass` | 将函数内显式 tensor 矩阵乘 lowering 成普通 Koopa 循环，并对固定内维展开 |
| `UnusedFunctionEliminationPass` | 从 `@main` 出发删除不可达函数 |
| `ConstantFoldingPass` | 折叠常量表达式 |
| `AlgebraicSimplifyPass` | 化简 `x+0`、`x*1`、`x*0` 等代数恒等式 |
| `GlobalValuePropagationPass` | 跨基本块传播稳定常量和别名 |
| `LocalLoadStoreForwardingPass` | 在基本块内转发局部标量 load/store |
| `GlobalScalarLoadForwardingPass` | 通过数据流分析跨块转发局部标量 load |
| `LocalDeadStoreEliminationPass` | 删除被覆盖且未读取的局部标量 store |
| `CommonSubexpressionEliminationPass` | 删除基本块内重复纯表达式 |
| `LoopInvariantCodeMotionPass` | 将循环不变量纯计算外提到 preheader |
| `BranchSimplifyPass` | 化简确定分支和同目标分支 |
| `JumpThreadingPass` | 穿透只包含跳转的中间基本块 |
| `UnreachableBlockEliminationPass` | 删除不可达基本块 |
| `DeadCodeEliminationPass` | 基于 IR 活跃变量分析删除无用赋值 |
| `UnusedAllocEliminationPass` | 删除已无使用的局部 `alloc` |
| `BlockCleanupPass` | 删除 terminator 之后的不可执行指令 |

后端使用 IR 层 `analyzeDenseLiveness` 的活跃变量分析结果构造 live interval，并采用线性扫描寄存器分配，不使用图着色。指令选择阶段还会使用 RISC-V 的立即数指令、`zero` 寄存器、移位替代乘 2 的幂等特性。

## 3. 实现情况

### 3.1 工作量与代码结构

项目代码按编译阶段组织：

- `src/compiler/lexer/`：词法规则、DFA 构造和 token 化。
- `src/compiler/parser/`：文法规则、预测分析表和 AST 构造。
- `src/compiler/ir/`：Koopa IR 生成、tensor IR 数据结构和 tensor 校验。
- `src/compiler/ir/opt/`：中端 IR 模型、parser、writer、pass manager、各类优化 pass。
- `src/compiler/ir/opt/tensor/`：tensor dialect 规整与融合逻辑。
- `src/compiler/ir/opt/passes/tensor_matmul_lowering.cpp`：tensor 矩阵乘 lowering。
- `src/compiler/riscv/`：RISC-V IR parser、栈帧、指令发射、窥孔优化和寄存器分配。
- `src/tests/`：单元测试，覆盖 lexer、parser、IR、优化、RISC-V 后端和寄存器分配。
- `manual_tensor_perf_cases/`：手写 tensor 和矩阵性能测试用例。

新增 tensor IR 后，没有把 tensor 作为注释外挂处理，而是纳入 lexer、parser、AST、Koopa 生成和 `IrModule`。在优化阶段，tensor 元信息和普通 scalar Koopa 指令保持分离；需要生成实际 RISC-V 的 tensor 矩阵乘会先 lowering 成普通 Koopa 控制流。

### 3.2 实现过程经验

本项目中比较重要的经验有三点：

1. 高层语义必须显式。普通数组循环可能只是普通二维数组程序，如果后端靠模式匹配猜矩阵乘或转置，容易误优化。因此矩阵类优化统一放到固定 tensor dialect 上。
2. 优化要用运行时间验证。多次尝试过普通数组转置识别、过度复杂的 RISC-V peephole、矩阵乘 RHS packing 等优化，其中部分在 qemu 下变慢，最后都回退或限制触发条件。
3. 寄存器分配应复用中端分析。项目已经有 IR 活跃变量分析，所以后端寄存器分配直接使用 `analyzeDenseLiveness`，避免再维护一套不一致的活跃分析。

### 3.3 AI 使用说明

项目开发过程中使用了 AI 辅助，主要 Agent 为 Claude 和 Codex。本仓库当前迭代主要使用 Codex 编程 Agent；本环境中的 Codex 是 GPT-5 系列代码模型。Claude 主要用于早期方案讨论、代码审阅和文档草稿辅助。

AI 主要参与环节：

- 梳理已有寄存器分配和优化 pipeline。
- 实现线性扫描寄存器分配、LICM、load/store 转发、DCE 等中端和后端优化。
- 设计 tensor dialect 的语法、校验、规整和测试。
- 编写 tensor 矩阵乘 lowering、专项 benchmark 和文档。
- 根据测试结果分析负收益优化并回退。

监督方式：

- 每次较大修改后运行单元测试或 Docker 性能测试。
- 对性能优化只采纳在 qemu RISC-V 程序运行时间上有收益的结果。
- 对 AI 生成代码进行人工审查，重点检查控制流、边界条件、未定义行为和是否误改无关文件。
- 对不稳定或负收益优化保留日志，确认后回退。

Token 和费用：

- 本地 Codex/Claude 会话没有统一导出完整 token 和费用账单，因此无法给出精确金额。
- 从开发过程看，AI 使用量属于多轮长上下文代码迭代，估计为数十万到百万级 token。
- 费用应以实际平台账单为准；本报告不把估算值作为精确费用。

经验教训：

- AI 很适合做代码迁移、补测试、跑实验和写文档，但优化方向必须用 benchmark 监督。
- 让 AI 一次性写太大的 pass 容易留下复杂死代码，最好按“分析空间、实现一小步、测试、再迭代”的方式推进。
- 文档需要在实现后同步更新，否则容易出现扩展说明与真实 pipeline 不一致的问题。

## 4. 达成效果

### 4.1 正确性

当前本地测试结果：

```text
157/157 tests passed
```

SysY 性能测试集结果：

```text
sysy-testsuit-collection: 467/467
```

Tensor 相关特性包括：

- 支持 `tensor a[2][3];` 作为源语言声明。
- Koopa IR 支持 `tensor %a: tensor<2x3>` 形状声明。
- 支持 `add`、`add_relu`、`broadcast`、`matmul`、`matmul_bias`、`matmul_bias_relu`、`matmul_relu`、`relu`、`reshape`、`transpose`。
- 支持矩阵乘 `_ta`、`_tb`、`_ta_tb` 转置视图变体。
- 支持函数内显式 tensor 矩阵乘 lowering 到普通 Koopa，再生成 RISC-V。

### 4.2 性能

全量 SysY 测试的历史迭代结果如下：

| 阶段 | 结果 | wall time |
| --- | ---: | ---: |
| 旧基线 `3d835cdabc5276cb92321d2c5e6358fbf1599733` | `455/467` | `1123.08s` |
| 线性扫描寄存器分配和 LICM 后 | `467/467` | `295.86s` |
| 加入跨块标量 load 转发与无用 alloc 删除后 | `467/467` | `247.49s` |
| 加入乘 2 的幂强度削弱后 | `467/467` | `243.69s` |
| 加入零比较专用发射后 | `467/467` | `239.67s` |

为了排除编译、汇编、链接和测试框架调度开销，项目也统计过 qemu 阶段的 RISC-V 程序运行时间：

| 阶段 | 结果 | RISC-V 运行时间 |
| --- | ---: | ---: |
| 历史最好后端 | `467/467` | `88.954647s` |
| 当前回退后工作区 | `467/467` | `89.664780s` |

两者差距约 `0.8%`，慢用例汇编基本一致，认为主要是 qemu 单次运行波动。

Tensor 矩阵乘专项 benchmark 使用两份语义相同的 `.koopa` 程序比较：

- `tensor_lowered`：显式 `tensor @C = matmul @A, @B`，由 tensor lowering 生成固定形状循环。
- `naive`：普通 Koopa `i-j-k` 三重循环。

测试只统计编译出的 RISC-V 程序在 `qemu-riscv32-static` 中的运行时间：

| 矩阵规模 | 重复次数 | 运行轮数 | tensor lowering 平均耗时 | naive 平均耗时 | 比例 |
| --- | ---: | ---: | ---: | ---: | ---: |
| `48x48` | `8` | `6` | `6.753ms` | `10.217ms` | `0.6610` |
| `96x96` | `2` | `4` | `8.091ms` | `13.656ms` | `0.5925` |

即 tensor lowering 相比普通三重循环约有 `1.51x` 到 `1.69x` 加速。主要收益来自固定内维展开、减少循环控制和栈访存、融合 bias/ReLU/transpose 视图，以及让寄存器分配更容易保留累加值。
